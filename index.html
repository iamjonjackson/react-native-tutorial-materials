Theory 10 min

Three approaches
to mobile development

"The client wants a mobile app. What do we build it with?"
Three distinct approaches.

🌐 Web & Wrap

Next.js + Vercel + PWABuilder · Ionic WebView

A web app — HTML, CSS, JavaScript — packaged and submitted to an app store. Still running inside a browser engine under the hood. The simplest path if you already have a web app.

⚡ Hybrid

React Native · Flutter · Ionic Capacitor

JavaScript (or Dart) logic drives genuinely native UI components — not a WebView. The platform renders real iOS and Android controls. One codebase, two native apps.

🔧 Native

Xcode + Swift · Android Studio + Kotlin

Platform-specific code, written in the platform's own language. Maximum performance and full access to every platform API. But you're maintaining two separate codebases.

K1 How organisations adapt and exploit digital technology solutions to gain a competitive advantage.

K5 A range of digital technology solution development techniques and tools.

Question

Has anyone used any of these approaches before, even in a personal project?

Theory 10 min

Pros, cons &
when to use each

Factors to consider

⏱ Time to market
🧑‍💻 Team skills
📱 Platform fidelity

🌐 Web & Wrap

Pros
  • Fastest if web app already exists
  • One team, one codebase
  • PWABuilder makes it straightforward
  • Minimal mobile-specific skills needed
Cons
  • Lacks native look or feel
  • Limited device API access
  • Animations feel "webby"
  • App stores may reject thin wrappers

✅ Best for: Content-led apps, internal tools, info apps where native feel isn't critical.

⚡ Hybrid (React Native)

Pros
  • Reuses React skills team may already have
  • Genuinely native UI components
  • One codebase for iOS, Android & web
  • Large ecosystem & community
  • Used by Meta, Microsoft, Shopify
Cons
  • Some native modules still require platform code
  • Build tooling can be complex
  • Debugging across the JS/native bridge
  • Not quite as fast as fully native

✅ Best for: Teams with React experience who need iOS + Android without doubling headcount. The sweet spot for many commercial projects.

🔧 Native (Swift / Kotlin)

Pros
  • Best possible performance
  • Full access to every platform API
  • First to get new OS features
  • Largest talent pool per platform
Cons
  • Two fully independent codebases to maintain
  • Increased team size and cost
  • Requires platform-specific expertise
  • Longer time to market

✅ Best for: Apps where the UI is the product — camera apps, games, AR/VR, financial trading. Worth the investment when scale demands it.

Theory 10 min

Publishing your app

Distribution costs and timelines can be easy to overlook...

🍎 Apple App Store

£79 / year *
Apple Developer Programme

Annual subscription. Covers Xcode access, TestFlight for beta testing with up to 10,000 testers, and App Store Connect for submissions. Apple's review process typically takes 1–3 days. Budget time for rejections — they happen, even to experienced developers.

🤖 Google Play

$20 one-off *
Google Play Developer Account

A single one-time registration fee. No annual renewal, no limit on the number of apps you publish. Google's review typically takes a few hours to one day. Still possible to be rejected — read the policies carefully.

🌐 PWABuilder

Free
pwabuilder.com

Microsoft's open-source tool that wraps a web app or Progressive Web App into packages for the App Store, Google Play, and the Microsoft Store. Good starting point for Web & Wrap projects. pwabuilder.com →

💰 Commissions

%
On paid apps & digital in-app purchases

Both stores take a cut of paid apps and in-app purchases of digital goods. Free apps pay nothing beyond the developer account fee. Physical goods (e.g. buying a product in a shopping app) are typically exempt.

* accurate as of June 2026

Live demo

Open pwabuilder.com and paste in a URL — show how quickly it analyses a site and offers store packages. Takes 60 seconds and makes the Web & Wrap approach concrete.

Theory 10 min

React Native tooling

The ecosystem has several layers. Knowing what each tool does — and when to use it — will save you hours of confusion.

⚡

Expo

A managed framework on top of React Native. Handles all the native build configuration so you don't have to. The recommended starting point for almost all new React Native projects.

Recommended
expo.dev →
📱

Expo Go

A phone app (iOS & Android) that lets you preview your project instantly by scanning a QR code. Perfect for learning — but limited for production-grade apps that need custom native modules.

Learning / Dev
🪄

Snack

React Native in the browser. Write code, see it run in a web player or Expo Go — no install. Every Snack gets a shareable URL. Great for demos, experiments, and teaching.

Zero Setup
snack.expo.dev →
☁️

GitHub Codespaces

A full VS Code environment running in the cloud — no local install needed. Fork a repo, open a Codespace, and you have Node, the Expo CLI, and everything else pre-configured.

Recommended today
🗂️

Expo Router

File-based routing for React Native and web, built on native navigation APIs. Drop a file in the app/ directory — it becomes a screen. Navigation automatically uses native patterns on each platform.

Today's nav approach
docs.expo.dev/router →
🚀

EAS (Expo Application Services)

Cloud builds and store submission from the terminal. When you're ready to publish, EAS compiles your app in the cloud and can submit directly to both stores. No Mac required for iOS builds.

When you publish
expo.dev/eas →
Theory 5 min

React Native vs
React DOM

If you know React for the web, you're 80% of the way there. Here's what's different.

Concept React (Web) React Native
Container <div> <View>
Text <p>, <span>, <h1> <Text>
Images <img src="…"> <Image source={…}>
Input <input> <TextInput>
Styling CSS files / modules StyleSheet.create({})
Layout Block, inline, flex Flexbox only (default)
Scroll Browser scroll <ScrollView>
React Native · App.tsx
import { View, Text, StyleSheet } from 'react-native';

export default function App() {
  return (
    <View style={styles.container}>
      <Text style={styles.heading}>
        Hello, mobile!
      </Text>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1,
    alignItems: 'center',
    justifyContent: 'center',
  },
  heading: {
    fontSize: 24,
    fontWeight: 'bold',
  },
});

The component model, hooks, state, props — all identical to React on the web. The primitives are just different.

Full VS Code in your browser. No local install needed.

How your phone connects (Tunnelling)

Your app runs in the cloud (Codespaces), but your phone still needs access.

What is tunnelling?

Tunnelling creates a public URL that connects your phone to your running app.

Phone → ngrok → Codespace → App
✓ Why we use it
  • Works on any network
  • No WiFi issues
  • Works with Codespaces
⚠️ Common issues
  • Slow loading → normal for tunnels
  • QR not working → open link manually
  • Fails → restart with tunnel mode

Hands-on 70 min

Supabase + Expo
user management app

Build a real app with Auth, a Postgres database, and file storage — all features you've used on the web, now running natively on your phone.

S4 Initiate, design, code, test and debug a software component for a digital and technology solution.

Follow the official tutorial: supabase.com/docs/guides/getting-started/tutorials/with-expo-react-native →

Phase 1 · ~20 min

Project setup — Supabase project, SQL schema, API keys, Supabase client config

  1. 1
    Create a new Supabase project Head to supabase.com/dashboard → New project. Choose a strong database password and save it somewhere.
  2. 2
    Run the starter SQL In the Supabase dashboard, go to SQL Editor → Quickstarts → User Management Starter This creates the profiles table with Row Level Security already configured.
  3. 3
    Copy your API keys Go to Project Settings → API. Copy the Project URL and the anon (publishable) key. You'll need these in your .env file.
  4. 4
    Install dependencies In your terminal: npx expo install @supabase/supabase-js @react-native-async-storage/async-storage react-native-url-polyfill
  5. 5
    Create lib/supabase.ts Initialise the Supabase client with your URL and anon key. Follow the tutorial exactly for the AsyncStorage session config.
Pause point

Make sure you have your EXPO_PUBLIC_SUPABASE_URL and EXPO_PUBLIC_SUPABASE_ANON_KEY in your .env file. This is a common place to get stuck — a wrong key will produce confusing errors later.

Phase 2 · ~25 min

Auth screens — magic link sign-in, session handling

  1. 1
    Build the Auth component An email input field and a "Send magic link" button. Call supabase.auth.signInWithOtp({ email }) on press.
  2. 2
    Handle sessions Use supabase.auth.onAuthStateChange() in a useEffect to update your session state when the user clicks the magic link in their email.
  3. 3
    Test on device Enter an email address in Expo Go. Check your inbox — the magic link should arrive within seconds. Tap it and confirm the session fires.
Phase 3 · ~25 min

Profile screen and avatar upload

  1. 1
    Build the Account component Fetch the user's row from the profiles table using their user ID. Display username, website, full name fields with a save button.
  2. 2
    Add avatar upload The Avatar component uses Expo's image picker to select a photo, then uploads it to a Supabase Storage bucket and saves the path to the profile.
  3. 3
    Test the full flow Sign in → view profile → update your name → upload an avatar photo. All running on a real device, backed by a real database.

Demo + Optional Code 20 min

Platform-specific
behaviour

This is React Native's headline argument over a pure web app. One codebase — two genuinely native experiences.

Part A — See the difference (5 min)

🍎 iOS

  • Back navigation: right-swipe gesture
  • Navigation headers use iOS typography
  • Haptic feedback on interactions
  • Sheet modals slide up from bottom

🤖 Android

  • Back navigation: system back button / gesture
  • Navigation follows Material You guidelines
  • Ripple effects on touchable elements
  • Dialogs follow Android conventions
Live demo

Expo example on Snack →. You scan the QR code on your iPhone or Android, then compare the difference with others who have a different type of device.

Part B — Optional: add it to your own app

If you've finished the Supabase tutorial, try adding platform-specific behaviour to your own codebase. If you're still finishing the tutorial, continue there instead.

Step 1 · Platform.OS — quick win

Add this to your Account screen to show which platform the app is running on:

import { Platform, Text, StyleSheet } from 'react-native';

// Inside your component's return:
<Text>Running on: {Platform.OS}</Text>

// Then use it to change styles per platform:
const styles = StyleSheet.create({
  container: {
    paddingTop: Platform.OS === 'ios' ? 20 : 10,
    backgroundColor: Platform.OS === 'ios'
      ? '#f2f2f7'
      : '#ffffff',
  },
});

Check both iOS and Android to confirm the value changes.

Step 2 · Tab navigation — stretch goal

Add Expo Router tab navigation to split your app into an Account tab and a new Home tab. The key files you'll need:

// File structure to create:
app/
  _layout.tsx        ← Root layout (already exists)
  (tabs)/
    _layout.tsx      ← Tab navigator config
    index.tsx        ← Home tab
    account.tsx      ← Move your Account screen here

Follow the structure at docs.expo.dev/tutorial/add-navigation →. The tab bar will automatically use native iOS or Android styling.

Inspiration ideas

Not a prescriptive menu — just a starting point. The most impressive demos usually come from ideas not on this list.

Approachable
